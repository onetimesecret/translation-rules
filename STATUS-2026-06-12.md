## Phase 2 — fan-out

**Goal:** bring every locale under governed rules, one at a time. This is when the system starts producing real value against the translation gap.

### Order (per SPEC §7)
1. **de** — already has `rules.yaml` as an inheritance parent; needs `register.yaml` and `glossary.yaml`
2. **pt_PT, uk, hu** — the other three contaminated by the 2026-04 incident; highest risk
3. **da_DK, zh** — lighter damage, next priority
4. **Remaining 28 locales** — mostly mechanical, locale → base with no intermediate tier

### Per-locale work (repeated for each)
* Author `register.yaml` with forbidden tokens — requires native-speaker confirmation
* Author `glossary.yaml` with sense disambiguation
* Author or update `rules.yaml`
* Set `baselines.yaml` entry
* Run `resolve.py <locale> --lint --emit=md,json`, verify green
* PR with native-speaker gate on the glossary

**Execution model:** `/d:work-tasks-db` parallel agents, one task per locale, sharded. Each task is self-contained. Native-speaker review is the human gate at merge for each locale.

### When translation work starts using the system
* **de_AT:** after P1-5b merges
* **Each other locale:** after that locale's Phase 2 migration PR merges

The 20% translation gap gets filled locale-by-locale, in the same order as Phase 2 migration. There is no big-bang moment. As each locale joins the governed system, the `saas-translator` skill can load its `.resolved/<locale>.json` instead of the inlined conventions table.
