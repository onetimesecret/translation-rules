## 1. Pre-YAML rough shape

First structural proposal in the thread, before the user countered with the `translation-rules/` YAML sketch. In-repo form.

> Rough shape, organized around the problems the retrospective surfaced:
>
> ```
> docs.onetimesecret.com/src/content/docs/<lang>/translations/
>   glossary.md              # prescriptive term rules, canonical
>   language-notes.md        # prescriptive style/register rules, canonical
>   baselines.md             # pinned commit SHAs per locale + rationale
>   _archive/                # non-normative history, clearly walled off
>     de-translation-notes.txt
>     <dated conversational reports>
> ```
>
> Key principles the structure enforces:
>
> **Prescriptive vs. descriptive is a directory boundary, not a tone choice.** `_archive/` is the firewall. Anything in `_archive/` can never be read as guidance; anything outside it must be prescriptive. The November 2025 mistake (promoting a change-log into the guide) becomes mechanically prevented: to promote, you have to physically move a file out of `_archive/`, which is a reviewable diff.
>
> **Baselines live in a dedicated file, not scattered in guide prose.** `baselines.md` is a table: locale → commit SHA → path → one-line "why this is the anchor". When someone asks "what does good de_AT look like", there's one answer, one file, one diff target. It doesn't decay into prose.
>
> **QA reviews feed back through a structured path, not as orphan reports.** The Nov 2025 review was a dead-end document. Proposal: `docs.onetimesecret.com/qa/locales/reviews/<date>/<locale>.md` stays for the raw findings, but each review must be accompanied by a PR touching `glossary.md` / `language-notes.md` / `baselines.md` with the corrections the review implies — or an explicit note saying "no guide changes needed". No review lands without a guide disposition.
>
> **The app project mirrors with generated artifacts marked as derived.**
>
> ```
> onetimesecret/locales/
>   content/<locale>/*.json     # the actual translations
>   guides/for-translators/      # GENERATED — do not edit
>     <locale>.md                # header: "Generated from docs project commit X"
>     .source-commit             # pinned upstream SHA
>   guides/SECURITY-TRANSLATION-GUIDE.md  # hand-authored, stays
>   guides/UX-TRANSLATION-GUIDE.md        # hand-authored, stays
>   baselines/                   # optional: per-locale revert anchors
>     de_AT.json.ref             # stub file with just "commit f95b03f44"
> ```
>
> The `# GENERATED` header + `.source-commit` pin is the loop-breaker. When a translation agent reads `de_AT.md`, it sees at the top: "Generated from docs project `abc1234`. Do not use as a source for future guide edits — upstream at docs.onetimesecret.com." That kills the assistant→guide→assistant round trip because agents reading the generated file know not to feed it back.
>
> **Per-locale override files, inheritance explicit.** `de_AT.md` should be a short override on top of `de.md`, not a parallel full guide. Structure:
>
> ```yaml
> ---
> inherits: de.md
> locale: de_AT
> ---
>
> # Overrides
> - formality: Sie (hard lock, never du)
> - email_example: kontakt@musterfirma.gv.at
> - ...
> ```
>
> Keeps the variant-specific rules visible and small, and makes "what's actually different about Austrian?" a one-screen answer rather than something you reconstruct by diffing two 400-line guides.
>
> **One thing the structure doesn't solve and the user should decide separately:** the generation pipeline itself. Right now I don't know whether `locales/guides/for-translators/de.md` is generated by a script, copied manually, or pulled through a submodule. Whatever the mechanism, it needs a regeneration gate: the docs project edit triggers an app project regeneration PR, and nothing ships until both land. Until the pipeline is explicit, the directory structure is necessary but not sufficient.
>
> ---
>
> **Status update on the parallel work:** neutral-editor finished — guide files in the docs project are edited (§1 rewritten as object/content split, change-log section removed, baseline pin added, archival banner on the .txt, regional tags on all examples). It flagged a few out-of-scope glossary inconsistencies (burn/löschen/verbrennen split, Eintragen vs. Anmelden, Bereich vs. Domäne) for QA to chase. ## 2. YAML tree elaboration

User prompt that introduced the `translation-rules/` YAML sketch:

> Something like this, that can mechanically produce the locale specific guide (inluding the UX and security gudielines):
>
> translation-rules/
> ├── base.yaml                    # Universal rules
> ├── de/
> │   ├── rules.yaml               # German grammar overlay
> │   ├── register.yaml            # Formality for DE
> │   └── glossary.yaml            # DE terms
> ├── de_AT/
> │   ├── rules.yaml               # AT overrides to DE rules
> │   ├── register.yaml            # AT formality (Sie, no exceptions)
> │   └── glossary.yaml            # AT-specific terms
> └── resolve.py                   # Merge script

Assistant response:

> The shape is right — this is the loop-breaker. Rules are hand-edited, outputs are generated, agents read outputs but can't write back to them. The only way new guidance enters the system is a human PR to a YAML file. That kills the assistant→change-log→guide→assistant round trip definitively.
>
> A few refinements and one thing the structure is missing:
>
> **base.yaml absorbs most of the current UX and security guides.** What reduces cleanly to rules (interpolation preservation, number format, "no emojis in UI strings", "don't translate brand names", placeholder semantics) becomes base.yaml keys. What doesn't reduce — rationale prose, brand voice, "why we translate error messages conservatively" — stays as `base/docs/*.md` and the resolver embeds those verbatim into output sections. Rules drive the lint; prose drives the reader. Both come from the same tree.
>
> **glossary.yaml needs to carry structured examples, not just term pairs.** The highest-leverage content in the current `glossary.md` and `language-notes.md` is worked examples — English source, translated target, context. Those need to be first-class data so the lint has teeth:
>
> ```yaml
> terms:
>   secret_object:        # the record/container
>     en: secret
>     de: Geheimnis
>     de_AT: Geheimnis
>     contexts: [burn, share, create, view, destroy, expire]
>   secret_payload:       # the revealed content
>     en: message
>     de: Nachricht
>     de_AT: Nachricht
>     contexts: [post_reveal, encrypted, truncated]
>
> examples:
>   - id: login_password_prompt
>     en: "Enter your password"
>     de: "Gib dein Passwort ein"
>     de_AT: "Geben Sie Ihr Passwort ein"
>     rule_refs: [register.de_AT.formality]
> ```
>
> Now `resolve.py --lint` can assert: every example in de_AT conforms to `register.de_AT.formality = Sie`; every string containing the English word `secret` in a burn/share/create context maps to `Geheimnis` not `Nachricht`; every interpolation `{var}` in `en` is preserved in targets. The rule tree becomes its own test suite.
>
> **register.yaml should be a hard enum, not prose.** For de_AT:
>
> ```yaml
> formality:
>   value: formal
>   pronouns: [Sie, Ihr, Ihnen, Ihre]
>   forbidden: [du, dein, dich, dir, deine, deinen]
>   rationale: "Austrian B2B default. Locked — see baselines.yaml commit f95b03f44."
>   exceptions: []   # explicit empty list, not missing
> ```
>
> The `forbidden` list is what the CI lint greps against `locales/content/de_AT/*.json`. Prose-only rules are how the April 2026 regression slipped past.
>
> **Inheritance semantics need to be explicit, not implicit.** "de_AT overrides de" is ambiguous — what happens to lists, to nested objects, to the `exceptions` array? Proposal: deep-merge for maps, replace for scalars, `merge:` strategy key on lists (`append | replace | prepend | dedup`). Put it in the resolver spec, not in tribal knowledge. The resolver needs its own tests for this.
>
> **Missing from your sketch — baselines and tests:**
>
> ```
> translation-rules/
> ├── base.yaml
> ├── base/docs/           # rationale prose the resolver embeds
> ├── baselines.yaml       # {locale: {commit: sha, path: ..., justification: ...}}
> ├── de/
> ├── de_AT/
> ├── resolve.py
> └── tests/               # resolver test fixtures
>     ├── inheritance/
>     └── lint/
> ```
>
> `baselines.yaml` is what my earlier structural proposal called `baselines.md` but in machine form — one commit pin per locale with justification. The resolver uses it for regression checks: "does the current locale content still agree with the baseline on any key that exists in both?" If not, flag the drift.
>
> `tests/` is the unglamorous necessity — the resolver has merge logic, the lint has assertions, both need tests. Without them the resolver becomes the new drift vector.
>
> **Dual-output resolver, not Markdown-only.** The resolver produces two things from one input:
>
> 1. Human-readable guides for translators: `docs.onetimesecret.com/src/content/docs/<lang>/translations/*.md` (Astro frontmatter, prose embedded from `base/docs/`) and `onetimesecret/locales/guides/for-translators/<locale>.md` (with `# GENERATED — source: translation-rules/ commit X` header).
> 2. Machine-readable merged data for tooling and agents: `onetimesecret/locales/guides/.resolved/<locale>.json`. This is what the CI lint reads. This is what a translation agent ingests for structured context instead of parsing Markdown prose.
>
> Same source, two audiences, zero copy-paste. The lint runs against the JSON, not the Markdown, so prose edits can't desync the rules.
>
> **Where the tree lives.** Probably in the onetimesecret repo, not the docs repo. The app is the authority on which strings exist; the docs site is downstream. `docs.onetimesecret.com` vendors the generated Markdown or pulls it from a published artifact. Inverts the current direction, which is part of how the loop formed — guide content flowed app-ward from a project that doesn't own the strings.
>
> **Open tradeoffs worth naming before committing:**
>
> - **YAML vs. JSON Schema + JSON vs. TOML.** YAML is human-friendly but whitespace-sensitive and has subtle parsing footguns (the Norway problem, string/number coercion). JSON Schema gives you free validation but the source files are noisier to edit. TOML handles multiline strings better than YAML and has no coercion surprises but no schema story. If the resolver is going to validate heavily, JSON Schema over YAML via a schema file is probably the sweet spot — edit YAML, validate against schema, reject malformed input at resolver time instead of lint time.
> - **Rationale prose in YAML vs. sidecar .md.** Short rationale in a `description:` key is fine. Long rationale (the kind currently in `language-notes.md` §"Regional Formality Considerations") should live in `<locale>/docs/*.md` and be referenced by path from the YAML: `rationale_doc: docs/formality.md`. Keeps YAML scannable and prose ergonomic to write.
> - **Migration cost.** Existing `glossary.md` and `language-notes.md` need to be parsed into YAML. One-time conversion; probably 3–5 hours of careful work because the source contradicts itself and you're making editorial judgments at every disagreement. The neutral-editor's reconciled output is a useful input here — it already resolved most of the contradictions.
> - **Agent ergonomics.** Translation agents currently consume `.md` guides. They'd need to consume the JSON output instead. Not hard, but it's a contract change — existing `d:translate-new-language` and `d:translate-parallel-agents` slash commands would need updating to feed the JSON in.
>
> **What the structure still doesn't solve.** The pipeline gate. Rules in YAML with a resolver is necessary but not sufficient — you still need CI that blocks any commit touching `locales/content/<locale>/` unless (a) the resolved JSON for that locale has been regenerated in the same commit, and (b) the lint passes against the new content. Without that gate, someone bypasses the rule tree by editing locale files directly and the drift resumes. The YAML structure is only as good as the CI that enforces it.
>
> ---
>
> **Practical note on the parallel work still running:** the general-purpose cross-locale audit is still in the background. I won't invoke qa-automation-engineer until it lands — the QA criteria need both outputs to validate against. Want me to hold the YAML architecture as a separate discussion thread, or fold it into the QA brief as "validate that the repaired guide content maps cleanly into this proposed structure"?

## 3. Retrospectives subtree

User prompt:

> We'll just write a spec.md for the Configuration Overlay Pattern when we are ready.
>
> How can the structure incorporate retrospectives?

Assistant response:

> Retrospectives are the missing piece — the November 2025 QA review was a retrospective that had no place to land, so its conclusions evaporated. The structure should make orphaning mechanically hard.
>
> **Shape:**
>
> ```
> translation-rules/
> ├── retrospectives/
> │   ├── README.md                              # format + template
> │   ├── 2025-11-16-de-docs-formality-drift.md
> │   ├── 2026-04-12-de_AT-register-flip.md
> │   └── _archive/                              # superseded / expired
> ```
>
> Each retrospective is a single file with machine-readable frontmatter and prose body:
>
> ```markdown
> ---
> id: 2026-04-12-de_AT-register-flip
> date: 2026-04-12
> triggered_by:
>   commits: [b08e59838, 4982d4f84, 44b3c3352]
>   incident: "de_AT formal→informal register flip"
> affected_locales: [de_AT]
> affected_rules:
>   - register.de_AT.formality
>   - terms.secret_object
>   - terms.secret_payload
> baseline_pins:
>   - locale: de_AT
>     commit: f95b03f44
>     path: src/locales/de_AT.json
> examples_added:
>   - login_password_prompt_de_at
>   - burn_secret_confirmation_de_at
> supersedes: [2025-11-16-de-docs-formality-drift]
> applies_until: null
> status: applied              # pending | applied | declined | superseded
> resolved_in_commit: <sha>
> ---
>
> # Summary
> # Evidence
> # Root cause
> # Disposition
> # Lessons not reducible to rules
> ```
>
> **Four things this buys you that a free-form retro doesn't:**
>
> **1. No orphaning.** CI refuses a retrospective PR with `status: pending` that doesn't either (a) touch the listed `affected_rules` / `examples_added` / `baseline_pins` in the same or a linked PR, or (b) flip to `status: declined` with a reason. The November 2025 failure mode — "useful analysis, nobody wired up the fix" — becomes a build failure. The retrospective can't exist in limbo.
>
> **2. Resolver attaches retro context to rule lookups.** When an agent (or a human) queries `register.de_AT.formality`, the resolver returns the rule *plus* every applied retrospective in `affected_rules`. The generated per-locale guide has an auto-built "History" section. Agents translating de_AT see: *"This rule is locked to Sie because of 2026-04-12-de_AT-register-flip. The previous drift was caused by X. Here's the pattern you must not reintroduce."* Institutional memory becomes part of the agent's prompt automatically, no one has to remember to paste it.
>
> **3. Baselines become retrospective-anchored.** Every entry in `baselines.yaml` must reference either a retrospective or an external rationale file. No orphan pins with forgotten justifications. If `de_AT: f95b03f44` exists in baselines, a retrospective somewhere explains why that commit and not another. Six months from now when someone asks "can we move this baseline forward?", there's a paper trail that either says yes (new retro supersedes) or no (the constraint that forced the pin is still live).
>
> **4. `declined` is a first-class status.** "We considered locking de_DE register after the November review but decided the informal voice is deliberate because [reason]" is as valuable as an applied decision. It prevents the next person from re-running the same analysis and arriving at the same wall. `_archive/` isn't where wrong retros go — it's where every retro eventually goes, preserved with its disposition.
>
> **Status lifecycle:**
>
> - `pending`: the retrospective exists, the implied changes haven't been made. CI warns on PR, blocks merge after a grace period (e.g., 7 days).
> - `applied`: the rule/example/baseline changes landed. `resolved_in_commit` is populated.
> - `declined`: the team considered and chose not to act. Reason required. Stays in `retrospectives/`, not `_archive/`.
> - `superseded`: a newer retrospective replaces it. Moves to `_archive/`. The newer one lists the old one in `supersedes:`.
>
> **Short-form retros matter.** Not everything is a multi-file analysis. A native speaker saying "this email phrase sounds weird" should fit the same template — one paragraph of evidence, one rule tweak, one example added. The frontmatter is the same; the prose body just happens to be three sentences. Keeping the format uniform across long and short retros means the resolver handles them identically.
>
> **Decay.** Not every past incident matters forever. `applies_until: <date>` on time-bounded learnings (e.g., "this workaround is needed until feature X ships"). `superseded_by:` for learnings that a later retro subsumes. The resolver filters expired/superseded entries out of agent context by default, but preserves them for traceability.
>
> **Cross-project retrospectives.** The November 2025 review was about the *docs* project but its root cause was a guide in the same ecosystem as the *app* project. Retrospectives need to be able to span both. Proposal: retrospectives live in the same place as the rule tree (inside the app repo, under `translation-rules/retrospectives/`), and the docs project vendors them through the same generation pipeline. One canonical home for institutional memory, two projects consuming it.
>
> **What retrospectives deliberately don't replace:** incident commit messages, PR descriptions, the git log itself. Those are the "what happened at this point in time" record. Retrospectives are the "what it means for future work" record. Related but different — the commit message for `b08e59838` said "Harmonize Austrian German locale"; the retrospective file says "this commit was the second wave of a five-month drift caused by change-log promotion and here's the structural fix that prevents recurrence". Don't conflate them; link them.
>
> **The specific value for agents.** Right now, a translation agent loading context for de_AT gets: English source + current guide Markdown. With retrospectives wired in, it also gets: every applied retrospective touching de_AT or any rule it's about to use, surfaced as negative examples ("don't produce this") and baseline anchors ("conform to this"). The agent's failure mode — "plausible local decision that contradicts a hard-won lesson" — becomes much harder because the lesson is literally in the prompt.
>
> **One thing this structure still doesn't give you and the spec should name:** retrospectives capture learning, they don't generate it. Someone has to notice something is wrong and file the retro. The November 2025 review existed because a QA pass ran; without the pass, no retro. The tree needs a *scheduled* QA mechanism — quarterly locale quality reviews, a cron that runs the lint against production content, a periodic native-speaker audit — to produce retrospectives on a cadence rather than only reactively. Retrospectives are the landing zone; the scheduled audits are the collection mechanism. Both matter. Separate concerns.
