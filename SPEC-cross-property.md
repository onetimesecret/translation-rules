# Translation Rules — Cross-Property Extension

**Status.** Proposal. Companion to [`SPEC.md`](./SPEC.md). SPEC v1 is scoped to the app repo (`SPEC.md` §5, §6). This document extends the same authority/resolver spine to the two supporting properties — the static website and the Astro/Starlight docs site — without rewriting the spine.

**Premise.** The quality regression that started this effort was diagnosed and the fix designed (`SPEC.md`). That fix covers one of three properties. The user's brief names all three — "application and supporting sites... each with a different way of working." The heterogeneity *is* the problem. This doc addresses it.

**Evidence basis.** All three properties were mapped from working clones (app: `onetimesecret`; docs: `docs.onetimesecret.com`; website: `onetimesecret.com`). The earlier draft's inferences about the two sites have been replaced with observed facts below.

---

## 1. Current state — the whole board

Three properties, three production pipelines, one shared body of linguistic knowledge currently maintained as prose.

| Property | Content format | Staleness mech. | Task tracking | Translation driver | Register surface |
|---|---|---|---|---|---|
| **App** (`onetimesecret`) | flat-key JSON `locales/content/<locale>/*.json` | per-key `content_hash`/`source_hash` | SQLite `translation_tasks` DB | `/d:start-translation-session` → ≤5 parallel `saas-translator` agents | UI strings (imperative + declarative) |
| **Docs** (`docs.onetimesecret.com`, Astro Starlight) | per-locale MDX trees `src/content/docs/<locale>/` (prose) + UI JSON `src/content/i18n/<locale>.json` | none — git diff by hand | none | `bin/translation-pluribus-util` prepare→combine→split→restore roundtrip; manual | long-form explanatory prose |
| **Website** (`onetimesecret.com`, Astro + vue-i18n) | nested JSON `src/i18n/ui/<locale>.json` + generated MD `src/content/pages/<locale>/*.md` | jq key-diff via `harmonize-locales.yml` GH Action | none | `vue-i18n-extract` scan/add/remove; manual translate; `generate-markdown-content.mjs` | **mixed** — UI strings *and* marketing prose |

Locale-coverage drift already visible: docs has 27 content locales but 17 configured in `config/i18n.mjs`; website has 23 content-page dirs but only 4 routing locales (en/fr/de/es).

**Shared knowledge — and where it actually lives (the live drift engine):**
- **Generic layer** — `saas-translator` skill (`~/.claude/skills/`): pluralization, date/number formats, formality *framework*. Hardcoded; reads no external rule file.
- **OTS guidance source** — physically lives in the **docs site repo** at `docs.onetimesecret.com/src/content/docs/en/translations/` (`guide.md`, `glossary.md`, `language-notes.md`, `universal/*`). `translation-rules/_references/en-translation-docs/` is a **copy** of it. The docs site also has a *second* machine-readable glossary at `docs/localization/glossary/core-terms.json`.
- **Per-locale** — `_references/local-guides/for-translators/<locale>.md`, generated by `bin/generate-for-translators` (present in both the docs repo and translation-rules) into `src/content/docs/<locale>/translations/`.

**Finding: there are three copies of the glossary** (translation-rules `_references/en-translation-docs/glossary.md`, docs site `en/translations/glossary.md`, docs site `localization/glossary/core-terms.json`) with no single source. This is not just a failure-vector risk — it is *active* drift. The cross-property authority's first job is to invert the current arrangement: today the docs site is the de-facto source and translation-rules imported from it; the target is translation-rules as source, with the docs site's `en/translations/` becoming a **generated** artifact.

**The failure vector, generalized.** The 2026-04-12 de_AT register flip happened because descriptive prose ("Thinking Behind Changes") flowed into a file an agent read as prescriptive. That vector exists on *all three* properties, because all three feed agents the same prose guides. SPEC's resolver closes it for the app. The website and docs site are still wide open.

**"harmonize = keys only" is concrete, not abstract.** SPEC Phase 0 §1 names this anti-pattern. It is the website's `harmonize-locales.yml` Action — jq that adds/removes *keys* to match `en.json` and preserves existing translation *text*. That automated version is the safe one. The original incident was a *manual* "harmonize" that rewrote text under the same word. The lesson for the authority: the only sanctioned harmonize is structural; text changes must come through the rule/retro path.

## 2. Unifying thesis

**`translation-rules` is the property-agnostic authority. Register, glossary, and rules describe the *language*, not a file format.** Each property keeps its own extract→translate→apply plumbing and consumes the same resolved artifacts through the same skill.

```
                    translation-rules  (authority: YAML + schema + resolver)
                              │
                resolve.py --emit=json  →  .resolved/<locale>.json   (+ for-translators/<locale>.md)
                              │
        ┌─────────────────────┼─────────────────────┐
        ▼                     ▼                     ▼
   App pipeline          Docs pipeline         Website pipeline
   (SQLite tasks)        (git-diff)            (git-diff)
        │                     │                     │
        └──────────── saas-translator skill ────────┘
            reads .resolved/<locale>.json — never hardcoded conventions,
            never the for-translators .md as rule source
```

This is the delta over SPEC §5/§6: SPEC already says the *app's* agents read `.resolved/<locale>.json` and the skill stops carrying hardcoded conventions. The extension is one sentence — **the docs and website agents read the same resolved JSON** — plus the surface-register split below, which is the only genuinely new modelling.

What does *not* change: the schemas, the resolver merge/inheritance/ID machinery (P1-1, P1-2 — both complete and tested), the retrospective lifecycle, the CI firewall. The spine is property-neutral already. Three consumers instead of one is a fan-out, not a redesign.

## 3. The one real modelling decision — surface register

Formality (Sie vs du, vous vs tu) is a pure language fact: it is identical across app, docs, and website. It belongs in `register.yaml` unchanged and shared.

But **voice/tone and some glossary senses are surface-dependent, not property-dependent.** Evidence already in the repo:
- `de.md`: "Nachricht" for UI messages vs "Geheimnis" in technical docs — same locale, different surface.
- `universal/voice-and-tone.md` is written entirely for UI (imperative buttons, declarative status). It has no model for long-form explanatory prose, which is most of the docs site.

The clean cut is **by surface type, not by repo** — the cut runs *through* the website, which carries both. Surfaces, not properties:

- `ui` — short interface strings; imperative actions, declarative status; tight character limits. (app; website `src/i18n/ui`; docs `src/content/i18n`)
- `prose` — long-form full sentences, no character limits. (docs MDX bodies; website homepage/about marketing copy)

A wrinkle the mapping surfaced: docs prose is *explanatory/neutral*, website prose is *marketing/persuasive* — arguably distinct registers. Resist splitting `prose` into `docs`/`marketing` until a concrete term actually conflicts between them (SPEC §8 discipline). Model it as two surfaces now; add a third only on evidence.

**Recommendation: add a `surface` facet to glossary senses and to voice rules, not a new top-level property dimension.** A glossary term carries its shared register plus optional per-surface sense overrides; the resolver emits both, and the consuming pipeline declares which surface it is requesting.

```yaml
# glossary.yaml term, surface-aware
- id: term.secret_object#7f3a91c2
  key: secret_object
  de: Geheimnis
  surfaces:
    ui:    { target: Nachricht,  rule_ref: glossary.de.ui-message-framing }
    prose: { target: Geheimnis,  rule_ref: glossary.de.prose-technical }
```

Resolver gains `resolve.py <locale> --surface=ui|prose` (default emits both partitions). The app requests `ui`; docs requests `prose`; the website requests both. Register/forbidden-tokens are emitted identically to every surface — formality is not surface-dependent.

This is the **only** schema change the extension requires. It is additive (terms without `surfaces` keep working) and stays inside SPEC §8's "extend the schema when the first case is touched" discipline. It should be deferred until the docs/website integration is actually scheduled — flagged here, not built now.

## 4. How each pipeline connects

The authority emits; pipelines consume read-only. Per-pipeline contract:

**App** — already specified (`SPEC.md` §6.1). Agents load `.resolved/<locale>.json` (surface `ui`); the drifted inline conventions table in `start-translation-session.md` (§6.4, already flagged as the same failure pattern) is deleted. No change to the SQLite task flow or hash staleness.

**Docs (`docs.onetimesecret.com`)** — the inversion happens here. Today `bin/generate-for-translators` reads guidance *from* this repo's `en/translations/` and writes `for-translators.md` back into it. Post-migration: the guidance source moves to translation-rules, and the resolver's Markdown emit (surface `prose`) produces `src/content/docs/<locale>/translations/for-translators.md`, hash-locked (SPEC §2.4). The `en/translations/` guide files become generated, not authored. `bin/translation-pluribus-util` (the prepare→split→restore roundtrip) is unaffected — SPEC §4 already keeps it; it shares the UUID/bundle patterns. Per-file translation agents load `.resolved/<locale>.json` for binding rules, never the generated guide as source. Consolidate the two stray glossaries (`en/translations/glossary.md`, `localization/glossary/core-terms.json`) into the authority's `glossary.yaml`; the JSON one becomes a resolver emit if anything still needs it.

**Website (`onetimesecret.com`)** — two surfaces from one consumer. Its UI JSON (`src/i18n/ui`) consumes `ui`; its marketing prose (homepage/about) consumes `prose`. Keep `harmonize-locales.yml` exactly as-is — structural key-sync is the *sanctioned* harmonize; the authority governs text, not keys, so the two don't collide. The only change: whatever/whoever fills in translated text (currently manual) loads `.resolved/<locale>.json` and treats generated guides as reference, never rule source. Reconcile the 4-routing-vs-23-content-dir locale gap as a side finding (file a backlog item; not blocking).

**Skill contract (`SPEC.md` §5) generalizes verbatim:** the skill reads `.resolved/<locale>.json` for binding rules, reads `for-translators/<locale>.md` only as human reference, writes nowhere upstream. Findings become human-gated `pending` retrospectives. This holds for all three consumers without modification — the skill doesn't know or care which pipeline invoked it, only which `surface` it requests.

## 5. The two genuine decisions — RESOLVED (2026-05-29)

Both decided per the recommendations below.

1. **DECIDED: inversion runs as the first act of Phase 1.5** (after P1-3 emit, before any per-locale docs translation), with a short docs-guidance freeze during cutover. — When does the guidance-source inversion happen? The docs site is currently the de-facto source of OTS guidance; translation-rules copied from it. As long as both exist, they drift (and already have — three glossaries). The inversion (authority becomes source, docs `en/translations/` becomes generated) is the single highest-leverage move, but it touches the docs repo and so can't precede the resolver's emit (P1-3). Recommendation: schedule it as the *first* act of Phase 1.5, before any per-locale docs translation — otherwise new docs translations are authored against a soon-to-be-stale source. The website's marketing register is the only thing that might force a third surface; defer that judgment to first marketing-copy conflict.

2. **DECIDED: `surface` facet lands at first prose migration** (first docs locale, Phase 1.5), not now. — Surface facet now or at first prose migration? SPEC's own discipline (§8) says extend the schema when the case is touched. The docs site is the first `prose` consumer, so the `surface` facet lands with the first docs locale, not before. The risk of waiting: the de_AT glossary authored in Phase 1 may need re-touching to add `surfaces:`. Low — formality dominates the de_AT incident, and formality is surface-neutral.

## 6. Sequencing relative to SPEC phases

No reordering of SPEC's phases. This extension attaches after Phase 1 proves the spine on de_AT app-side:

- **Phase 1 (unchanged):** de_AT end-to-end, app only. Proves resolver + emit + CI.
- **Phase 1.5 (this doc):** (a) **invert the docs source** — move `en/translations/` guidance into the authority, consolidate the three glossaries, make the docs guide files a resolver emit; (b) first docs locale (de or de_AT) consumes `prose`-surface resolved output. Introduces `--surface`, retires `bin/generate-for-translators`. One locale, one property — reversibility checkpoint intact.
- **Phase 2 (unchanged):** fan-out to 34 locales via `/d:work-tasks-db`. Each per-locale ticket now emits both surfaces; docs + app consume in the same pass.
- **Phase 3 (website):** website joins. Lower lift than feared — it's already mapped, and its `harmonize-locales.yml` stays untouched. It consumes both surfaces (`ui` from `src/i18n/ui`, `prose` from marketing copy). Or stays frozen.

## 7. What this does not solve (additions to SPEC §9)

- **Inversion is a one-time cutover with a window.** While the docs source is being moved into the authority, both copies exist; a translation authored in that window can be based on the wrong one. Keep the window short and freeze docs-guidance edits during it.
- **Surface misclassification.** If a string is mis-tagged `ui` vs `prose`, it gets the wrong glossary sense. Lint can assert a term's surfaces are mutually exclusive; it cannot know which surface a given app key actually renders into. That stays human judgment at authoring.
- **Three diff mechanisms persist.** This doc unifies *guidance*, not *plumbing*. App keeps hash-staleness + SQLite; docs keeps the pluribus roundtrip; website keeps `harmonize-locales.yml`. Unifying the plumbing is a separate, larger effort and explicitly out of scope — the shared authority is the high-leverage win; shared transport is not.
- **Locale-set drift across properties is unmanaged.** App ~35, docs 27-content/17-configured, website 23-content/4-routing. The authority can define which locales *have rules*; it does not force any property to ship them. That mismatch stays a per-property decision.
